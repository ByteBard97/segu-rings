01HERO

Loop time is a ring.

A guitar practice app with a looping track. Loop time is a circle. Every operation in the UI on the left — where am I in the bar, how close is the next chord, snap to nearest beat, fade in the upcoming chord — is a cyclic question.

The library doing the math underneath is segu. It’s a well-designed linear-units library. The README says so on the first line: simplified conversions and transforms for linear units. Linear is the assumption.

Most of these operations work most of the time. The ones that break, break at the loop boundary, and the fix is always the same: stop doing math on the line, start doing math on the circle. Five places that matters, scene by scene.

For the geometry behind why this works, see “Your angles are lying to you.”

02PROGRESS · invlerp
where am I in the bar?

Sawtooth at the wrap.

The bar timeline highlights beats as the playhead advances. The natural code is progress = (now − barStart) / barLength. In segu, that’s Units.invlerp.

segu/src/math.js
// Interpolation function returning the ratio
// of a value clamped between x and y
export function invlerp (value, x, y) {
  return clamp((value - x) / (y - x))
}
naive segu naive
const progress = units.invlerp(now, { min: 0, max: barLength })
your code hand-rolled
const progress = ((now - barStart) % barLength) / barLength
on the circle lifted
const θ = 2 * Math.PI * now / barLength
const progress = [Math.cos(θ), Math.sin(θ)]

Modulo arithmetic doesn’t make the space cyclic; it makes the storage cyclic. The math still happens on the line.

03DISTANCE
how far to the next chord?

The shortest distance question.

To fade in the next chord’s display, the app needs to know how many beats away it is. The natural code is nextChord − now. Negative across the wrap, positive everywhere else. The fade-in fails to fire on the seam.

naive subtraction naive
const delta = nextChord - now   // negative on wrap
double-modulo hand-rolled
const forward = ((nextChord - now) % L + L) % L
// always positive — but it answers FORWARD distance,
// not SHORTEST distance.
arc on the circle lifted
const a = unit(now),  b = unit(nextChord)
const arc = Math.atan2(cross(a,b), dot(a,b)) * L / (2*PI)

The hand-rolled version isn’t broken; it’s incomplete. It answers “forward distance” when the question was “shortest distance.” The lift makes the question well-defined.

04LERP
cross-fade between two loop positions.

Halfway around the wrong side.

When transitioning between two loop positions, the natural code is lerp(t, a, b). The canonical formula. Correct for linear values. Disastrous for cyclic ones — halfway between 7.8s and 0.2s is supposed to be 8.0s (the seam), not 4.0s (the opposite side).

segu/src/math.js
export function lerp (ratio, x, y) {
  return (x * (1 - ratio)) + (y * ratio)
}
naive segu naive
units.lerp(0.5, { min: 7.8, max: 0.2 })  // → 4.0s
special-case wrap hand-rolled
function loopLerp(t, a, b, L) {
  if (Math.abs(b - a) > L / 2) {     // magic constant
    if (b > a) a += L; else b += L
  }
  return ((a*(1-t) + b*t) % L + L) % L
}
slerp the vectors lifted
function loopLerp(t, a, b, L) {
  const [ax,ay] = unit(a), [bx,by] = unit(b)
  const [x,y] = [lerp(t,ax,bx), lerp(t,ay,by)]
  return project(x, y, L)   // atan2 back to scalar
}

The lift doesn’t add complexity, it removes complexity. The hand-rolled lerp had a magic constant and a special case. The lifted lerp doesn’t.

05SNAP
snap to nearest beat.

The fence around storage.

The lit dots in the bar timeline snap to beat positions. The natural code is Math.floor(now / beat) * beat. In segu, that’s Units.snap. Pure linear rounding — no knowledge that the value lives on a loop.

segu/src/units.js (excerpted)
snap (value = 1, { to = this.lens.unit, calc = Math.floor } = {}) {
  const unit = this.normalize(to)
  const output = calc(value / unit) * unit
  return this.normalize(output || 0)
}
naive segu naive
units.snap(now, { to: beatLength })   // can land past barLen
re-wrap with modulo hand-rolled
(Math.round(now / beat) * beat) % L
// round(7.9) → 8 → %8 → 0. It works.
// Until you switch from round to floor and it doesn't.
nearest of N points lifted
const a = now / L * 2 * PI
const step = 2 * PI / N
return Math.round(a / step) * step   // stays on the circle

The loop has no edge, so snap doesn’t either. The modulo wrapper was a fence around a problem that didn’t need a fence.

06CYCLIC
but isn’t that what cyclic is for?

cyclic() is storage, not math.

The fair objection. segu does have a cyclic function. It folds an out-of-range value back into range. It does not make the resulting value behave cyclically when used in subsequent math.

segu/src/math.js
// Cycles a value around an range (from x to y).
// Useful for condensing overflowed values into a wrappable scope,
// such as animation loops.
export function cyclic (value, x, y) {
  return (value >= x ? value : value + y) % y
}
walk it through
cyclic(7.9, 0, 8)            // → 7.9   (already in range)
0.1 - 7.9                    // → -7.8  (the bug is back)

The space is still a line. cyclic did nothing wrong; the next operation re-introduced the bug.

The architectural point: segu’s math.js is pure — no unit awareness. That purity is correct; it’s what makes the library clean. But it means there is no place in the architecture for “this value lives on a circle, not a line.” Cyclic math has no home in segu. It has to live somewhere else.

The lift isn’t a fix to segu. It’s a sibling abstraction — linear units and cyclic quantities are different shapes, and they deserve different libraries.

segu’s tagline is “simplified conversions and transforms for linear units.” The library is honest. The architectural gap is between linear units and cyclic ones, and that gap is where every hand-rolled modulo expression in your codebase lives. The lift fills the gap.

07CLOSING

What the lift removes.

Five operations. Five demos. Same shape every time.

  • naiveexploded at the boundary.
  • hand-rolledmostly worked — with a magic constant (L/2), a special-case if, a double-modulo, or a sign-of-delta gotcha.
  • liftedthree to five lines, no special cases, no magic constants.

The lift’s selling point isn’t that it fixes a bug. It’s that it deletes a layer of code. Every modulo expression, every shortest-distance helper, every if (delta < 0) delta += L — those exist because the value was being stored as a number when it should have been stored as a point.

If your codebase has a cyclic quantity — loop time, day of week, heading, phase, hue, longitude — and you find yourself reaching for %, that’s the signal. Embed the value in 2D. Do the math up there. Project back when you need a scalar to display. Most of the time you don’t.
