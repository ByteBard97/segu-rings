The ring patch.

What if segu understood cyclic quantities natively? Here is the smallest set of changes that makes the library ring-capable — a new module, six new methods on Units, two new properties on Lens, and five helpers in math.js.

The rule: never do scalar arithmetic across a seam. Lift to the circle, do the math in ℝ², project back with atan2. The library stays pure; it just gains a second gear.

← back to the demos · essay · the ring patch

The obvious fix is to wrap values with modulo: (a + b) % period. This works for addition, but it fails for interpolation, distance, mean, variance, snap, and every other operation that assumes a continuous line. Worse, each operation needs a different modulo hack, and the hacks don't compose. The vector approach is different: one lift, one project, and your existing linear math works unchanged because it's operating in the right geometry.

01math.js

Five helpers for the lift.

The original math.js is pure linear: lerp, invlerp, cyclic. The ring-aware additions live in the same file, use the same style, and call the existing lerp internally. No breaking changes.

segu/src/math.js
export function lerp (ratio, x, y) {
  return (x * (1 - ratio)) + (y * ratio)
}

export function invlerp (value, x, y) {
  return clamp((value - x) / (y - x))
}

export function cyclic (value, x, y) {
  return (value >= x ? value : value + y) % y
}

export function steps (ratio, all) { /* … */ }

/* ========== Ring math — lift, operate, project ========== */

const TAU = 2 * Math.PI

export function toCircle (value, period = 1) {
  const a = (value / period) * TAU
  return { x: Math.cos(a), y: Math.sin(a) }
}

export function project (point, period = 1) {
  const a = Math.atan2(point.y, point.x)
  const t = ((a / TAU) + 1) % 1
  return t * period
}

export function ringDistance (a, b, period = 1) {
  const av = toCircle(a, period), bv = toCircle(b, period)
  const cross = av.x * bv.y - av.y * bv.x
  const dot   = av.x * bv.x + av.y * bv.y
  return Math.atan2(cross, dot) * period / TAU
}

export function ringLerp (ratio, a, b, period = 1) {
  const av = toCircle(a, period), bv = toCircle(b, period)
  const x = lerp(ratio, av.x, bv.x)
  const y = lerp(ratio, av.y, bv.y)
  return project({ x, y }, period)
}

export function ringMean (values, period = 1) {
  const n = values.length
  const sx = values.reduce((s, v) =>
    s + toCircle(v, period).x, 0) / n
  const sy = values.reduce((s, v) =>
    s + toCircle(v, period).y, 0) / n
  return project({ x: sx, y: sy }, period)
}
02lens.js

Declare the period.

Lens already carries min, max, and grid. Two additions: period (the loop length) and isRing (a convenience flag). When period is set, the lens knows the quantity lives on a circle, not a line.

    unit = 1,
    is = 1,
    as = 1,

  } = {}) {

      unit, is, as,
      min, max, grid, origin,
      period
    }
  }

  get unit () { /* … */ }
  /* … is, as, min, max, grid, origin … */

  get period () {
    return this.data.period
  }

  get isRing () {
    return this.data.period != null
  }
}
03units.js

Ring-aware methods on Units.

The existing Units class stays untouched for linear work. Six new methods wrap the math.js helpers and pull period from the lens. If period is absent, the methods fall back to max - min.

segu/src/units.js
import { Lens } from './lens'
import {
  gcd, cyclic, clamp,
  lerp, invlerp,
  ringLerp, ringMean,
  ringDistance, toCircle,
  project
} from './math'

export class Units {
  /* … lerp, invlerp, snap, cyclic … */

  /* ===== Ring-aware methods ===== */

  toCircle (value, lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    return toCircle(value, period)
  }

  project (point, lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    return project(point, period)
  }

  ringLerp (ratio, a, b, lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    return ringLerp(ratio, a, b, period)
  }

  ringMean (values, lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    return ringMean(values, period)
  }

  ringDistance (a, b, lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    return ringDistance(a, b, period)
  }

  ringSnap (value, {to = this.lens.grid || 1} = {},
             lens = this.lens) {
    const period = lens.period ||
      (lens.max - lens.min) || 1
    const steps = period / to
    const p = this.toCircle(value, lens)
    const theta = Math.atan2(p.y, p.x)
    const snapTheta = Math.round(
      theta / (2*Math.PI) * steps) *
      (2*Math.PI) / steps
    return this.project({
      x: Math.cos(snapTheta),
      y: Math.sin(snapTheta)
    }, lens)
  }
}
04ring.js · new file

A standalone Ring type.

If you prefer objects to scalar wrappers, Ring stores the lifted vector directly. It mirrors the Cyclic class from the essay, but uses the same toCircle / project helpers as the rest of segu.

segu/src/ring.js new
export class Ring {
  constructor (value, min = 0, max = 1) {
    this.min = min
    this.period = max - min
    const t = this._norm(value)
    const a = 2 * Math.PI * t
    this.x = Math.cos(a)
    this.y = Math.sin(a)
  }

  _norm (value) {
    const p = this.period
    return (((value - this.min) % p) + p) % p / p
  }

  get value () {
    const a = Math.atan2(this.y, this.x)
    const t = ((a / (2 * Math.PI)) + 1) % 1
    return this.min + t * this.period
  }

  distanceTo (other) {
    const cross = this.x * other.y - this.y * other.x
    const dot   = this.x * other.x + this.y * other.y
    return Math.atan2(cross, dot) * this.period / (2 * Math.PI)
  }

  static lerp (ratio, a, b) {
    const x = (1 - ratio) * a.x + ratio * b.x
    const y = (1 - ratio) * a.y + ratio * b.y
    return Ring._fromXY(x, y, a.min, a.period)
  }

  static mean (rings) {
    const n = rings.length
    const sx = rings.reduce((s, r) => s + r.x, 0) / n
    const sy = rings.reduce((s, r) => s + r.y, 0) / n
    return Ring._fromXY(sx, sy, rings[0].min, rings[0].period)
  }

  static _fromXY (x, y, min, period) {
    const a = Math.atan2(y, x)
    const t = ((a / (2 * Math.PI)) + 1) % 1
    const r = new Ring(min, min, min + period)
    r.x = Math.cos(a)
    r.y = Math.sin(a)
    return r
  }
}

export const ring = (value, min, max) =>
  new Ring(value, min, max)
05Usage

What the API looks like.

Same lens, two modes. Linear when period is absent; ring-aware when it is present. The calling code does not change shape — only the result.

your code
import { Units } from 'segu'

const units = new Units({
  lens: { min: 0, max: 8, period: 8 }
})

// lift to the circle — now a 2-vector, no seam
const p = units.toCircle(now)

// distance (shortest, signed)
units.ringDistance(now, nextChord)

// lerp across the seam — the hard case
units.ringLerp(0.5, 7.8, 0.2)  // → 0.0  (shortest path)
// linear lerp would give: 4.0  (long way around — wrong)

// mean of angles clustered near 0
units.ringMean([7.8, 0.2, 1.5]) // → ~0.47
// arithmetic mean would give: ~3.17  (wrong)

// snap to nearest beat on the ring
units.ringSnap(now, { to: beatLength })

The patch is small.

One new file, six new methods, two new lens properties. Nothing existing changes. The linear path stays the default; the ring path is opt-in via period.

The linear path stays the default. You only pay for ring math when you set period on the lens. The overhead is two trig calls per value (toCircle) and one (project) per result — negligible for UI animations, game logic, and most real-time use cases. The API doesn't change shape; the same methods return correct answers instead of wrong ones.

The architectural point: cyclic quantities are not a special case of linear ones. They are a different shape. The library that handles both is not a bigger linear library — it is a library with two gears.
